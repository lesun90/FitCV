# Cleanup Report — FitCV

## Summary
Overall the codebase is healthy: `strict` mode is on, there are zero `any` types, error handling is consistent (try/catch + `caught instanceof Error` everywhere), and the domain layer is cleanly separated from UI. The dead weight is small and concentrated. Top 3 issues: (1) a handful of confirmed unused symbols/imports that `tsc --noUnusedLocals` flags (`patch`, `Settings` import effectively decorative, `ReactNode` in LatexEditorRoute, the `active` param in `renderLayoutControlEditor`, `TemplateId` in templateAdapters, and dead destructured props `checks`/`warningCount` in `EditorWorkspace`); (2) ~22 orphaned CSS classes in `styles.css` (a `paper-*` block, `stepper`/`swatches`/`template-switcher`, etc.) that no component references — likely leftovers from a removed style/template-switcher panel; (3) `App.tsx` at 1755 lines mixes the `App` container, `Dashboard`, `EditorWorkspace`, all seven section editors, and ~12 pure helper functions in one file — several self-contained clusters are cleanly extractable.

## Priority 1: Correctness Risks
- No `any` usage anywhere in `src/` (verified). No loose typing introduced beyond unavoidable Vite-glob / `JSON.parse` casts.
- `src/ui/App.tsx:385,391` — `EditorWorkspace` destructures props `checks` and `warningCount` that are never used in its body (the dashboard uses them, the editor does not). `tsc --noUnusedLocals` flags both. Not a runtime bug, but dead props that misrepresent the component's contract; safe to drop from the prop type and the `EditorWorkspace(...)` call site (App.tsx:259-278).
- `src/ui/App.tsx:1005-1009` — `renderLayoutControlEditor(module, active, onChange)` never reads `active`; flagged by the compiler (TS6133). The parameter can be removed along with its two call sites' argument (only one real caller at line 991).
- `src/services/latexCompiler.ts:168` — `(entry as { texmflog?: string }).texmflog` reaches into the BusyTeX `CompileResult` shape that the published types do not expose. This is a controlled cast (not `any`) but is the one spot relying on an undocumented field; if the dependency changes shape it silently yields `undefined`. Worth a comment, not a fix.
- Error-handling consistency is good, with one asymmetry: `src/ui/App.tsx:202-211` `importArchive` has no try/catch, so a malformed `.fitcv` (e.g. the secret-field rejection thrown in `archive.ts:52`) surfaces as an unhandled rejection and leaves `busy` stuck on "Importing archive". Every sibling async handler (`importPdf`, `compile`, `exportArchive` indirectly) wraps errors and clears `busy` in `finally`. Recommend matching that pattern.

- `src/ui/App.tsx:24` and `:458` — the `Settings` lucide import drives a single non-functional button (`<button className="chrome-icon" aria-label="Editor settings"><Settings /></button>`) with no `onClick`. Either wire it up or remove the button + import.
- `src/ui/LatexEditorRoute.tsx:1` — `type ReactNode` is imported but never used (TS6133). Remove from the import.
- `src/domain/templateAdapters.ts:3` — `TemplateId` is imported but never used (TS6196). Remove from the type import (the line keeps `TemplateKey`, etc.).
- `src/ui/styles.css` — 22 class selectors with zero references in any `.tsx` (verified individually): `compact-list`, `compile-action`, `control-row`, `design-card-head`, `log-block-compiling`, `module-name` (note: only `.module-name-input` is used; the bare `.module-name` rules at lines 1629 and 1678 are orphaned), `paper-contact`, `paper-content-section`, `paper-header`, `paper-list`, `paper-role-block`, `paper-role-head`, `paper-role-sub`, `preview-head`, `preview-toolbar`, `resume-paper`, `selectable`, `stepper`, `stepper-unit`, `swatches`, `template-switcher`, `unit`. The `paper-*` cluster and `resume-paper` look like a removed full-page paper preview; `stepper`/`swatches`/`template-switcher`/`unit` look like a removed style controls panel. Safe to delete after a quick visual confirm.
- No commented-out code blocks found anywhere in `src/`.

## Priority 3: Simplification
- `src/domain/latex.ts:82-86` and `src/domain/templateAdapters.ts:295-299` — `visibleProfileHighlights` is duplicated verbatim (same body, same fallback to summary split). Extract one shared helper (e.g. in `domain/resume.ts` or a small `domain/profile.ts`) and import in both. Note `App.tsx:1677 profileHighlightsForResume` is a near-cousin but returns full `ProfileHighlightItem[]` with synthetic ids rather than trimmed strings, so it is related but not identical — fold carefully if at all.
- Space-clamp logic is duplicated: `clampSpaceValue` at `templateAdapters.ts:291` (`Math.min(96, Math.max(0, Math.round(value*10)/10))`) is re-implemented inline at `App.tsx:1013`. Export `clampSpaceValue` (and the `96`/`0` bounds) from `templateAdapters.ts` and reuse it in `renderLayoutControlEditor`.
- Magic constant: the space max `96` and default `12` appear in `templateAdapters.ts` (`defaultSpaceValue = 12`, clamp `96`) and again hard-coded across `App.tsx` (`value: 12` at line 666; `min={0} max={96} step={1}` and the `Number.isFinite(...) ? : 12` default at lines 1011, 1025-1044). Promote `defaultSpaceValue` and a `MAX_SPACE_VALUE`/`MIN_SPACE_VALUE` to named exports and consume them in the UI sliders/number inputs so the bounds can't drift.
- The seven section editors (`ExperienceEditor`, `EducationEditor`, `ProjectsEditor`, `CustomSectionsEditor`) share an identical card scaffold: `subhead` + Add button, then `.item-card` with grip / hide-toggle / delete-toggle head. The head block (grip, title, hide button, delete button) is copy-pasted 4×. Extract an `<ItemCardHead>` (and possibly a generic `<ListSection>` wrapper) to remove ~80 lines of repetition.
- `ProfileTextField` (App.tsx:1470) and `ProfileNestedTextField` (App.tsx:1502) are 90% identical (same row markup, same hide toggle); they differ only in the onChange writer. Could be unified with a single `onChange` callback prop.

## Priority 4: Structure
`src/ui/App.tsx` (1755 lines) — clearly extractable, self-contained units (no closures over `App` state; they already take all data via props/params):
- `Dashboard` + its children `ResumeGroup`, `FitCard`, `FilterItem`, `MiniPaper`, `TopChrome`, and the `formatRelative` helper → a new `ui/Dashboard.tsx` (~lines 282-377, 519-600, 1603-1617, 1749-1755). Fully prop-driven.
- All seven section editors plus their pure update helpers (`renderSectionEditor`, `SummaryEditor`, `ExperienceEditor`, `EducationEditor`, `ProjectsEditor`, `SkillsEditor`, `AwardsEditor`, `CustomSectionsEditor`, `Field`, `ProfileTextField`, `ProfileNestedTextField`, and the `editField`/`updateProfileField`/`updateExperience`/… family at lines 1620-1747) → a new `ui/sectionEditors.tsx`. These are module-level pure functions/components with zero `App` closure deps.
- `StylePanel` (lines 602-972) is large and self-contained (drag-and-drop layout editor) → `ui/StylePanel.tsx`.
- `PreviewPanel` (lines 1524-1601) → `ui/PreviewPanel.tsx`.
- The module-level helpers `labelForLayoutModule` (55) and `updateLayoutModule` (63) and `sectionLabels` (43) are shared by StylePanel and editors — move to `ui/layoutModules.ts` (or co-locate with the layout types) so the extracted files can import them without a circular dep back into `App.tsx`.
Doing the Dashboard + sectionEditors + StylePanel extractions alone would cut `App.tsx` to roughly the `App` container + `EditorWorkspace` shell (~400-500 lines).

## Skip list
- `src/domain/latex.ts` (`renderLatexSource`) and the structured-PDF fallback branch in `src/services/pdf.ts:69-105` look unused because only `awesome-cv` (the only template with an adapter) is selectable in the editor. They are NOT dead: `services/importer.ts:15` creates `classic-ats` resumes from PDF import, and archive import can carry `classic-ats`/`modern-compact` records — both flow through the non-adapter PDF path and `renderLatexSource`. Keep.
- `src/ui/App.tsx:882-937` — the `StylePanel` non-adapter branch (`usesLayoutModules === false`, plus the `reorder`/`moveSection` helpers it uses) is effectively unreachable today because `EditorWorkspace` (lines 424-428) force-switches any non-`awesome-cv` resume to `awesome-cv`. It is a defensive fallback for templates without adapters; do NOT delete without a product decision, since it is the only section-order editing path and re-enabling `classic-ats`/`modern-compact` in `visibleLayoutTemplates` would resurrect it.
- `validateTemplateRegistry` (`templates.ts:202`), `busyTexLicenseReview` (`latexCompiler.ts:24`), `detectMainTexCandidates`/`buildLatexFileTree`/`getLatexFileKind` (`latexProject.ts`), `extractPdfText` (`pdf.ts`), `createUuid` (`ids.ts`) — all appear "export-only" from a single direction but are genuinely consumed (tests, `LatexEditorRoute`, `latexTemplates.ts`, `importer.ts`, `createId`). Keep.
- `ReviewMarker.sourceSnippet` (`types.ts:102`) is set by `importer.ts` but never read in the UI. It is part of the persisted record schema (`schemaVersion: 1`) and asserted in tests; treat as data-model surface, not dead code. Keep.
- `LatexFileKind = 'unsupported'` / the `binaryExtensions`-vs-`textExtensions` split — all branches are reachable via `latexTemplates.ts` filtering and tests. Keep.
- The various `as` casts (Vite `import.meta.glob` results, `JSON.parse` in `archive.ts`, `structuredClone` re-typing in `resume.ts`, `bytes.buffer.slice(...) as ArrayBuffer` in `pdf.ts`) are necessary, localized, and correctly narrowed. Do not "clean up" into `any` or remove.
